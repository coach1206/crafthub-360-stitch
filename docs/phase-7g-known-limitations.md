# Phase 7G — Known Limitations (Honest, Carried Forward + New)

## Carried forward from prior phases

- **POS/E.A.T. reference images (Phase 7D/7E):** The two reference images for the POS3 handheld and E.A.T. command-center redesign were shared inline in chat, not as repo file uploads. No matching binary was found anywhere in the repo after an exhaustive search. They are preserved as text-described visual direction in `public/design-references/phase-7/pos-eat/README.md`, with an explicit disclaimer that binaries were not available. The redesign was built from that text spec plus direct visual inspection — not from a saved image asset.
- **DayOne360 has only one real screen (Phase 7F):** `src/pages/DayOneTravel.jsx`, mounted at `/dayone360-travel` and `/dayone360`. No dedicated `dayone/`, `travel/`, `concierge/`, or `vendor/` folder structure exists. All "DayOne360 routes" language in later phase instructions refers to this single screen plus its real backend (`server/routes/travelRoutes.js`).
- **DayOne360 service tiles disabled pending backend workflow (Phase 7F):** Airport Pickup, Housing Support, Private Driver, SIM & Phone Setup, Bank Setup Guidance, Local Area Tour, Business Introductions, Vendor Matchmaking, Emergency Support, and Document Checklist are all rendered but disabled with specific reasons ("Booking gateway not connected", "Concierge workflow not connected yet", "Vendor system not connected") because no backing service exists for any of them yet.
- **Payment gateway not fully connected where applicable:** POS3 checkout flows use the existing local `paymentService.js`/`CheckoutDrawer` — there is no live third-party payment processor integration. This was true before Phase 7 and remains true; Phase 7 did not change payment processing.
- **Some POS/E.A.T. advanced tools disabled where no backing workflow exists:** Table QR scanning and staff messaging (POS3Handheld); Floor Plan editor, Table Assignment workflow, and Reservations (EATCommandHub) are all visibly present but explicitly disabled with accurate reasons, matching the reference-image structure without fabricating functionality.

## New limitations identified during Phase 7G QA

- **No live browser/device testing was performed in Phase 7G.** The responsive QA (`docs/phase-7g-responsive-qa.md`) is a static code-level review of layout primitives (`clamp()`, `flexWrap`, `minmax()`, fixed `maxWidth`) — it is not a substitute for testing in an actual browser at real viewport widths, or on a physical handheld POS device. This is the single largest gap before any production claim.
- **EATCommandHub's `Live Ops Feed`/`Active Alerts` two-column grid (`1.7fr 1fr`) has no stacking breakpoint** and may feel cramped on a narrower tablet held in portrait. Not fixed in this phase (counts as a new redesign, out of scope for a QA-only phase) — flagged as a follow-up.
- **`components/commandhub/CommandHub.jsx` (NOVEE OS Command Hub) showed zero hits for the responsive primitives (`@media`, `clamp()`, `minmax()`, `flexWrap`) checked elsewhere** — it was built primarily for desktop/admin use and was not verified for tablet/handheld rendering in this pass. Recommend a manual pass if tablet/handheld admin access is ever expected.
- **POS3Handheld and DayOneTravel are intentionally capped at a narrow `maxWidth`** (480px and 680px respectively) by design, since they are handheld/mobile-first screens. On a wide desktop browser they render as a centered narrow column rather than a full desktop layout. This is expected behavior for their stated purpose, not a defect — but is worth knowing if either screen is ever expected to double as a tablet checkout surface.
- **One decorative-only button was found and fixed in this phase:** the POS3Handheld header notification bell had no `onClick` and was not marked `disabled`. It is now honestly disabled with the reason "Notification center is not yet built." No other decorative buttons were found across the Phase 7A–7F screens audited.
- **SmokeCraft.jsx's "View Passport" inner row (~L309) has its own `cursor:pointer` styling but no independent `onClick`** — it currently works because it sits inside a parent card that does have the handler, but this is fragile to a future refactor. Not fixed in this QA-only phase (low risk, works correctly today) — flagged as a follow-up.

## Scope boundary note

Per the explicit Phase 7G instruction, this phase performed QA, documentation, and only "safe, low-risk polish" — no new redesigns, backend rewrites, or route architecture changes were made. The one functional code change in this phase (the POS3Handheld notification bell) falls within the allowed "button disabled-state clarity" polish category.

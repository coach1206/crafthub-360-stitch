# MVP2 Global Visual Image Registry

This is the source of truth for every uploaded reference image and every
runtime image used by an image-based screen, across all families
(SmokeCraft, POS3, E.A.T., Passport, DayOne360, CraftHub).

## Hard rules (do not violate)

- **Build passing is not visual approval.**
- **Grep passing is not visual approval.**
- **Verification script passing is not visual approval.**
- **Claude saying "looks aligned" is not visual approval.**
- **The rendered proof image (`docs/visual-proof/<screen>-proof.png`,
  generated by `node scripts/visualProof.js`) is the only thing that can
  move a row below from `planned`/`in progress` to `visually approved`.**
- A reference image is a **visual specification** (composition, spacing,
  density, colors, card proportions, background framing, button
  placement, feature layout, premium look, responsive behavior) — it is
  **not** automatically a runtime background. Full UI mockups must be
  rebuilt with React/CSS; only a verified clean-photo region may be used
  as a runtime asset, and never stretched beyond its native aspect ratio.
- Never guess a reference-to-route mapping. If the mapping is unconfirmed,
  the row below says so and the screen is wired with `reference: null` in
  `scripts/visualProof.js` until a human confirms it. (This is the lesson
  from the golden-box route mistake: `GOLDEN BOX JOURNEY11.png` was
  initially miswired to `/smokecraft/golden-box` instead of the correct
  `/smokecraft/golden-box/status`.)

## Status legend

- `planned` — screen/route exists or is scoped, no reference-driven build work done yet.
- `in progress` — UI work has started but no passing proof image yet.
- `visually failed` — a proof image was generated and the rendered side does not match the reference.
- `visually approved` — a proof image was generated and the rendered side matches the reference.
- `blocked — no reference file in repo` — only a text description exists; no binary image to compare against. Visual proof is impossible until a real file is uploaded.
- `blocked — mapping unconfirmed` — one or more candidate reference files exist in `public/` but are not wired into any route in `src/`, and the correct mapping has not been confirmed by a human.

## POS3 (handheld + tablet/stationary)

A real reference image (`POS 3.11.png`) was uploaded 2026-06-25 and copied
to `public/design-references/mvp2/pos3/POS 3.11.png`. It is wired to
`/pos3/handheld` (confirmed via `src/App.jsx`: `/pos3/handheld` ->
`src/pages/pos3/POS3Handheld.jsx`) in `scripts/visualProof.js`.

**Important finding from running the proof harness:** `/pos3/handheld`
(and every other `/pos3/*` route) is gated behind a staff-login
`ProtectedRoute` (`requiredPermission="access_pos3_staff"`, `demoBlocked`).
Playwright hits this route unauthenticated, so the rendered capture is an
"ACCESS RESTRICTED" gate screen, not the actual POS3 Handheld UI. The
harness now detects this (`accessGated: true` in the proof JSON) and
prints a warning instead of treating the proof image as a real
comparison. No visual judgment can be made about POS3 Handheld until the
harness can authenticate as a staff user before capturing.

## Maintenance

- Every time `node scripts/visualProof.js` (or a `visual:<family>` /
  `visual:all` npm script) is run and a new `<screen>-proof.png` is
  generated and visually inspected, update the relevant row's Status
  column to `visually approved` or `visually failed` — never to
  `visually approved` based on build/grep/script output alone.
- When a new reference image is uploaded, add a row immediately, even if
  every other column is `TBD`, so it is never silently dropped.
- When a Passport/DayOne360/CraftHub mapping is confirmed by a human,
  update both this table and the corresponding entry in
  `scripts/visualProof.js` (change `reference: null` to the real
  filename) in the same commit.

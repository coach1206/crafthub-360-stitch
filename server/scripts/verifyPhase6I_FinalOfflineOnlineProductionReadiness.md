# Phase 6I — Final Offline/Online Production Readiness Manual Verification

These are the live-execution steps for every row marked **MANUAL** in
`phase6I_final_venue_grade_test_matrix.md`. No jest/vitest/Playwright
harness exists in this repo (confirmed again this phase, consistent with
every prior phase's verification doc) — these require a real running
backend bound to a migrated Postgres database, and a real browser
session. They are not automated here; `runPhase6IReadinessChecks.js`
covers everything that can be proven from static source inspection alone.

## 1. Fresh build

1. `npm run build` — must complete with 0 errors. Record module count and
   any warnings.

## 2. App and backend start

2. Start the backend (`node server/index.js` or equivalent npm script) —
   confirm it binds to its port without throwing, and that
   `DATABASE_URL` is set to a real migrated Postgres instance.
3. Start the frontend dev/build server and load the app in a browser —
   confirm no console errors on initial load.

## 3. Offline browser simulation

4. In DevTools → Network, set throttling to "Offline". Create a POS3
   order. Confirm the UI does not hang or throw, and the event appears in
   `SyncStatusPanel.jsx`'s pending/queued count (staff view) without any
   "synced" claim.
5. Inspect IndexedDB (Application tab → IndexedDB → the sync queue
   database/object store used by `syncQueueService.js`) — confirm the
   queued event row exists with the correct `eventId` and payload, and
   is not marked confirmed.

## 4. Reconnect / retry

6. Re-enable network (set throttling back to "Online" / "No throttling").
   Confirm `syncQueueService.js`'s retry-on-reconnect logic fires within
   its normal interval and POSTs the queued event(s) to
   `/api/sync/events`.
7. Confirm the IndexedDB queue row is removed (or marked confirmed) only
   after the POST returns a real success response — not optimistically
   before the network call resolves.

## 5. Backend confirmation

8. After step 6, confirm the resulting `sync_events` row in Postgres has
   a `backendConfirmationId` (or equivalent confirmed marker) set to a
   value generated by `recordEvent()`, not echoed from any client
   request body field.

## 6. Duplicate protection

9. Replay the same `eventId` a second time via `POST /api/sync/events`
   (or `POST /api/sync/replay`) — confirm `conflictType:
   'duplicate_event_id'` → `already_processed_no_action`, never
   re-applied.
10. Submit a second event with a different `eventId` but the same
    `businessActionFingerprint` for a sensitive event type (Order/
    Checkout/Payment/Stamp/Passport/Ticket/Humidor) — confirm
    `manual_review_required`. Repeat for a non-sensitive event type and
    confirm `server_rejected_duplicate`.

## 7. Replay preview and request

11. As staff, in E.A.T. Command Hub, trigger a replay preview
    (`POST /api/sync/replay/preview`) on a flagged event — confirm no
    `sync_events` row is mutated (compare `replay_status` before/after)
    and only `sync_audit_logs`/`sync_event_lifecycle` rows are written.
12. Trigger an actual replay request (`POST /api/sync/replay`) — confirm
    the resulting `backendConfirmationId` matches a real `recordEvent()`
    return value and the UI reflects the real outcome, not an optimistic
    one.

## 8. Reconciliation

13. From `SyncConflictReviewPanel.jsx`, add a reconciliation note to a
    flagged event — confirm 400 on an empty note, success on a valid one,
    and that the note is attributed to the signed-in staff member.
14. Resolve a reconciliation item with a staff-authored `staffReason` —
    confirm success, and confirm that attempting to resolve by sending
    only a client-supplied `backendConfirmationId` (no `staffReason`) is
    rejected.

## 9. Audit timeline

15. Load the audit timeline for a known `eventId` via
    `SyncAuditTimelinePanel.jsx` — confirm it renders the actor/device/
    role history described by `sync_audit_logs`.
16. Load the audit timeline by `businessActionFingerprint` — confirm it
    correctly groups all related events.

## 10. Staff-only access / guest denial

17. As a signed-in staff user, confirm access to `/eat` and all three
    staff sync/audit panels succeeds.
18. As a guest (no session) or non-staff role, confirm `/eat` is blocked
    by `ProtectedRoute`'s `requiredPermission="access_eat_command"`, and
    that direct `curl` calls to any `/api/sync/reconciliation*`,
    `/api/sync/conflicts*`, `/api/sync/audit*` route return 401 (no
    session) or 403 (non-staff session) — never 200.

## 11. Degraded mode

19. Stop the backend or unset `DATABASE_URL` and hit
    `/api/sync/audit/summary` — confirm `{dbAvailable:false,
    degraded:true, message}`, never a fake empty success.
20. With the backend reachable but unauthenticated, confirm the frontend
    health pill in `SyncAuditTimelinePanel.jsx` shows "Sign-In Required"
    (401) vs "Staff Access Required" (403) vs "Backend Unavailable"
    (network/5xx) as three visually distinct states.

## 12. No false claims

21. Across all three staff sync panels, confirm no instance of
    "Fixed"/"Recovered"/"Fully resolved"/"Fully synced"/"Fully processed"
    appears anywhere a real user would see it (header comments
    documenting what NOT to claim are excluded).
22. Confirm no panel renders raw JSON as its primary view — only behind
    an explicit "Show technical details" toggle.

## 13. Final go/no-go decision

23. Once steps 1–22 are executed and their results recorded, use the
    outcomes — together with the static evidence in
    `phase6I_final_venue_grade_test_matrix.md` and
    `runPhase6IReadinessChecks.js` — to make the final go/no-go call in
    `PRODUCTION_LOCKDOWN_PHASE_6I_FINAL_PRODUCTION_READINESS_REPORT.md`,
    Section L. Any step above that fails or cannot be executed (e.g., no
    live Postgres instance available in this environment) must be listed
    as a known limitation, not silently assumed to pass.

## Why these remain manual

Running steps 1–22 requires a live Express server bound to real sockets,
a migrated Postgres database, real JWT issuance, and a real browser
session with DevTools network throttling and IndexedDB inspection. None
of this infrastructure exists in this repo as an automated harness (no
jest/vitest/Playwright config, no test database bootstrap script) —
consistent with every prior phase (6B–6H) reaching the same conclusion.
Per the task's own guidance, these are documented as manual checks rather
than forced into a brittle automated script.
